## 摘要
视觉-语言-动作模型、世界模型和智能体规划器各自推动了物理智能的发展，但它们的组合缺乏统一的执行抽象、共享状态、语义验证以及跨异构载体的持久经验。我们提出PhyAgentOS，一个运行时基础架构，将调度、验证、记忆、基准测试和安全性作为系统级服务提供。其以会话为中心的运行时将会话（而非单个动作）视为调度、兼容性预检、受控执行、证据收集和验收的最小单位。为了将认知与物理执行解耦，认知-物理边界是一个文件系统：状态即文件协议将跨层状态具体化为带YAML的Markdown格式，生成可检查、可版本化的记录，且代理层与运行时层之间无代码依赖。这些视图形成了一个统一的认知状态空间，协调意图、能力、环境、执行和经验。会话验证器通过基于证据的成功、失败或重规划判断，区分执行终止与语义任务完成。验证后的结果通过认知记忆整合为可复用知识和纠正性经验，形成一个无需重新训练的试错闭环。基准测试复用部署会话和验证路径，使结果可追溯至实际执行。分层安全机制约束策略驱动和代理驱动的执行：预检、动作桥接、安全守卫、心跳监控以及目标局部约束。验证是渐进式的：游戏测试认知规划，模拟添加动力学和控制，真实机器人添加硬件噪声，同时保持认知层不变。PhyAgentOS在Optimus-67、StarDojo和DST-Dojo上进行了基准测试，并在19种以上的模拟和物理载体上得到验证，在LIBERO、Calvin和RoboCasa365上使用多种VLA模型取得了性能提升。

## Abstract
Vision-language-action models, world models, and agentic planners each advance physical intelligence, yet their composition lacks a common execution abstraction, shared state, semantic verification, and persistent experience across heterogeneous embodiments. We present PhyAgentOS, a runtime foundation delivering scheduling, verification, memory, benchmarking, and safety as system-level services. Its Session-Centered Runtime treats a session, not an action, as the minimum unit of scheduling, compatibility preflight, supervised execution, evidence collection, and acceptance. To decouple cognition from physical execution, the cognition-physics boundary is a file system: the State-as-a-File protocol materializes cross-layer state as Markdown with YAML, yielding inspectable, versionable records without code dependencies between Agent and Runtime layers. These views form a unified cognitive state space aligning intent, capabilities, environment, execution, and experience. The SessionVerifier distinguishes execution termination from semantic task completion via evidence-grounded verdicts of success, failure, or replan. Verified outcomes are consolidated through epistemic memory into reusable knowledge and corrective lessons, closing a trial-and-error loop without retraining. Benchmarking reuses the deployment session and verification path, so results trace to real execution. Layered safety constrains both policy-driven and agent-driven execution: preflight, action bridges, SafetyGuard, heartbeat monitoring, and target-local constraints. Validation is progressive: games test cognitive planning, simulation adds dynamics and control, real robots add hardware noise, with the cognitive layer held constant. PhyAgentOS is benchmarked on Optimus-67, StarDojo, and DST-Dojo, validated on 19+ simulated and physical embodiments, and gains on LIBERO, Calvin, and RoboCasa365 across multiple VLA models.

# 论文总结：PhyAgentOS：一种自我演化的具身智能体操作系统

## 1. 核心问题与整体含义（研究动机和背景）

- **核心问题**：当前的具身智能系统在处理“拿杯子”这类任务时，存在一个结构性缺陷：**系统缺乏一个用于验证物理任务是否完成的中介层**。视觉-语言-动作模型（VLA）可能成功执行了轨迹，世界模型预测了状态，智能体规划器报告了无错误调用，但最终物体并未被拿起。这种“**验证鸿沟**”导致执行成功与任务完成被混为一谈。
- **背景与动机**：
    - **三大范式各自为政**：VLA模型、世界模型和Agent系统三大研究范式各自取得了重要进展，但缺乏统一的执行抽象、共享状态表示和语义验证机制。当它们被简单组合时，会因抽象层级不匹配、晦涩难懂和缺乏持久记忆等问题而失效。
    - **现有机器人操作系统不足**：机器人操作系统（ROS）主要提供消息传递中间件，而不具备任务级调度、跨会话持久记忆、语义验证和统一安全架构等上层操作系统服务。
    - **整体含义**：需要一个**操作系统级的基础架构**，位于上述三大范式之下，提供调度、验证、记忆、安全等系统级服务，从而解决“验证鸿沟”问题，实现认知与物理执行的解耦，并建立自我进化的闭环。

## 2. 方法论：核心思想、关键技术细节

**（1）核心思想**：
- **解耦认知规划与物理执行**：将系统分为**Agent层**（决定“做什么”）和**Runtime层**（决定“如何做”）。Agent不直接控制硬件，而是生成结构化的“会话”。
- **会话为中心**：将会话（Session）作为任务调度、执行、验证的最小单元，而非单个原子动作。
- **文件系统即边界**：将认知-物理边界视为一个文件系统。通过“**状态即文件**”协议，所有跨层数据都以人类可读的Markdown（嵌入式YAML）文件形式呈现。

**（2）关键技术细节**：
- **Session-Centered Runtime（会话中心运行时）**：
    - **WatchdogSupervisor（看门狗监控器）**：负责从`SESSIONS.md`中认领会话，进行预检，监控心跳，并写回结果。它是会话生命周期的管理者。
    - **SessionRunner（会话执行器）**：具体协调选定的 `SkillRuntime` 和 `Target`，管理观察-动作循环。
    - **SkillRuntime（技能运行时）**：提供特定的执行逻辑，包括**Policy-driven Execution（策略驱动）**（用于VLA等连续控制）和**Agent-directed Tool Execution（Agent导向工具执行）**（用于Agent在线决策）。
- **State-as-a-File Protocol（状态即文件协议）**：
    - 核心文件包括：`SESSIONS.md`（任务契约与历史）、 `SKILLRUNTIME.md`（执行方法描述）、 `TARGETS.md`（目标对象属性）、 `ENVIRONMENT.md`（环境状态）、 `KNOWLEDGE.md`（成功经验）、 `LESSONS.md`（失败教训）。
    - 这些文件构成了统一的**认知状态空间**，将任务意图、运行时能力、环境状态、执行状态和历史经验对齐。
- **SessionVerifier（会话验证器）**：
    - 接收执行证据，并根据会话契约（任务目标、初始状态、最终状态、执行轨迹等）进行判断。
    - 返回三种结论：**success（成功）**、**failure（失败）** 或 **replan（重规划）**。它替代了简单的“动作完成”信号，实现了**语义验收**。
    - 判断函数可形式化为：`V(G, S₀, S_T, τ, H) → {success, failure, replan}`，其中`G`是任务目标，`S₀`和`S_T`是初始和最终环境状态，`τ`是执行轨迹，`H`是历史上下文。
- **Epistemic Memory（认知记忆）**：
    - 验证后的成功和失败经验被分类并存储在 `KNOWLEDGE.md` 和 `LESSONS.md` 中。
    - 系统通过 **Execute → Verify → Diagnose → Revise → Re-verify → Consolidate** 的闭环，积累可复用的知识和纠正性教训，实现**自我演化**，**无需重训神经网络**。
- **Layered Safety Architecture（分层安全架构）**：
    - 包括：兼容性预检、动作桥接（Action Bridge）、SafetyGuard（安全守卫）、心跳监测和目标局部约束五层，形成递进式的安全边界。

**（3）算法流程（文字说明）**：
1.  **Agent层**：接收用户请求，利用上下文、记忆和工具，生成一个或多个结构化的“会话”并写入`SESSIONS.md`。
2.  **Runtime层**：`WatchdogSupervisor` 发现新会话，进行兼容性预检。通过后，创建一个 `SessionRunner`。
3.  **执行**：`SessionRunner` 协调 `SkillRuntime` 和 `Target`，进行观察-动作循环，直到会话终止。
4.  **验证**：`SessionVerifier` 收集执行证据，与任务目标对比，给出语义结论（成功、失败或重规划）。
5.  **记忆与进化**：验证结果被 `Epistemic Memory` 吸收到知识库或教训库中，影响未来Agent的决策，形成闭环。

## 8. 不足与局限

- **轮询式协议引入延迟**：当前文件协议基于轮询机制，会引入与轮询间隔成比例的延迟。虽然设计上保证了健壮性，但在对延迟敏感的实时应用中可能需要事件驱动的替代方案。
- **长期记忆缺乏压缩边界**：`LESSONS.md`和`KNOWLEDGE.md`会随着时间推移无限制增长。当前的选择启发式（任务类型匹配）有效，但缺乏理论上的保证，且自动化的技能质量保证仍是半手动。
- **真实机器人评估还不够全面**：虽然支持了19种机器人，但关于真实机器人的大规模、系统化任务完成统计数据有限，更多集中在对安全机制的验证上。
- **规划图编译（Goal Graph & Session Compiler）尚未完全实现**：论文架构中描述的DAG级任务分解与跨载体的依赖追踪功能尚未完成，限制了其在需要多机器人协作的复杂场景下的应用。
